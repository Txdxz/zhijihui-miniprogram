## 1. 系统概述

"智机惠"是一款面向中国移动各省用户的合约办理与权益核销微信小程序。系统通过与中国移动权益平台小程序对接（经世纪恒通中转），实现"合约办理 → 用户在【河北移动权益优选】小程序领取权益 → 世纪恒通调用智机惠 `/order/create` 生成代金券 → 门店核销"的完整闭环。代金券由世纪恒通调用智机惠 `/order/create` 接口（mobileIntegration 云函数）触发生成（而非合约触发），生成即激活（status=1，当月有效），支持用户主动核销和自动结算两种模式。系统面向各省移动公司，接口对接模式一致。

**客服联系方式：** 客服电话 4007071506（服务时间 09:00-17:30 周一至周五）

系统包含四类核心角色：

| 角色 | 标识 | 主要职责 |
|------|------|----------|
| 客户 | `customer` | 办理合约、领取权益、到店核销 |
| 门店店员 | `store_clerk` | 门店核销代金券 |
| 门店店长 | `store_owner` | 门店核销（权限等同店员） |
| 管理员 | `admin` / `super_admin` | 合约审核、办理、发货、系统管理 |

**异常分支说明：**
- `status=0` 拒绝：管理员核验未通过，客户可重新提交
- `status=8` 验证码被驳回：客户需重新输入中国移动下发的验证码，回到 `status=3`

**status=5 自动注册说明：**

当合约状态到达 `status=5`（合约已办）时，系统自动执行 `bindPhoneToUser` 操作，将客户提交的手机号与当前微信 openId 绑定，完成用户注册。注册原理如下：

- 合约办理时，客户已提交中国移动下发的短信验证码并通过管理员验证，手机号真实性有保障
- 系统利用已验证的手机号，在 `users` 表中创建或更新 `openId` → `phone` 的映射关系
- 注册完成后，用户下次使用同一微信登录时，`resolveLaunchContext` 可直接识别其身份，无需重复办理即可查看权益状态和卡包

> **注意：** 代金券不再由合约触发生成。代金券统一由世纪恒通调用智机惠 `/order/create` 接口（mobileIntegration 云函数）时生成，生成即激活（status=1，当月有效）。

**关键接口：**
- `submitPhone`：提交手机号，创建合约记录
- `submitOrderInfo`：提交收货信息
- `submitSmsCode`：提交短信验证码
- `bindPhoneToUser`：合约办理完成后自动绑定手机号到 openId，完成用户注册
- 管理员侧：核验、办理、发货、签收相关接口

**关键数据表：** `contracts`、`coupons`、`users`

### 2.3 中国移动权益对接流程（含代金券生成与自动结算）

**流程说明：** 用户符合领取条件后，在【河北移动权益优选】小程序点击"领取"，世纪恒通（第三方平台）调用智机惠 `/order/create` 接口（mobileIntegration 云函数）生成代金券。代金券生成即激活（status=1），当月有效（有效期至当月最后一天 23:59:59）。系统在生成代金券时随机设定结算时间，支持用户主动核销和自动结算两种模式。

**自动结算机制（懒触发）：**

代金券在生成时随机设定一个结算时间 `settleAt`（生成后 1~25 天内的随机白天时间 9:00-18:00）。系统在以下时机进行懒触发检查：

- 用户查询卡包时
- 用户生成核销码时

检查逻辑：若当前时间 >= `settleAt` 且该券尚未结算（`settleType` 为空），则使用 CAS（条件更新）进行并发保护，调用 `notifyHengtongSettle`（经 mobileIntegration 云函数）向世纪恒通上报 `verifyStatus=1`（已核销状态），并记录 `settleType='auto'` 和 `settleAt` 结算时间。用户前端不受影响，券仍显示为"可使用"（status=1），代金券在自动结算后仍可继续使用，当月过期后前端显示"已过期"（status=3）。

**结算回执回调说明：**

结算回执回调地址为世纪恒通在 `/order/create` 请求中传入的 `notifyUrl`（**不再直接回调中国移动**）。回调报文采用 3DES 加密，并附带 MD5 签名校验，确保报文安全性与完整性。

**用户使用流程说明：**

- **权益内容：** 宠物门店服务类商品代金券一张
- **领取权益：** 用户订购权益成功后，在【河北移动权益优选】小程序页面，点击"我的-我的订单"或点击链接 `https://dx.10086.cn/0oqmu6Rz` 至微信小程序打开，找到订单页点击【去使用】，点击优惠券即可领取优惠

1. 代金券自领取到账之日起当月有效，当月未领取自动失效，不可转赠他人，请在有效期内使用，过期不补

4. 适用门店：猫庄宠物用品超市(众创大厦店)、裕华小甄宠物用品店、猫庄·宠物生活馆、小鹿宠物(盛邦花园六区店)、有猫腻宠物生活馆、芊宠萌宠物服务、安淇宠物医院（秦皇岛店）

**客服电话：** 4007071506（服务时间 09:00-17:30，周一至周五）

**对接要点：**
- 智机惠**不通过接口同步给移动平台**，用户自行前往【河北移动权益优选】小程序领取
- 智机惠**不提供**跳转到【河北移动权益优选】小程序的链接，仅做文字提示引导
- `/order/create` 接口需校验世纪恒通签名（3DES 加密 + MD5 签名），防止伪造回调

- `/order/create`（mobileIntegration 云函数）：世纪恒通调用生成代金券，返回券信息（`cardNo`, `cardDeadline` 等）

**核销码规则：**
- 6 位数字
- 有效期 3 分钟，过期需重新生成
- 二维码内容为核销码本身（或带签名的 URL）

**卡包查询机制（手机号匹配）：**

```
用户登录 → 查询 portal_users 或 users 表获取 phone
    │
    ├─→ 有手机号 → 按 phone 查询 coupons 表

    └─→ 无手机号 → 引导用户手机号授权（getPhoneNumber）

```

手机号获取优先级：`portal_roles` 绑定的手机号 > `users` 表记录的手机号 > 微信授权获取的手机号。

**关键接口：**
- `generateVerifyCode`：生成核销码

- `storeConfirmVerify`：确认核销（需传入 `verifyCode` 参数进行安全校验）

### 2.5 管理员邀请绑定流程

**流程说明：** 超级管理员通过手机号邀请管理员或门店负责人，被邀请人首次打开小程序时通过微信官方 `getPhoneNumber` 授权完成手机号解密与角色绑定。该流程是微信官方推荐的"企业小程序管理员邀请绑定"标准做法。

```
                └─────────────────────────────┘
                              │
                              ▼
                ┌─────────────────────────────┐
                │  被邀请人打开智机惠小程序     │
                │  （通过微信分享/扫码进入）    │
                │  ★ 此时微信仅获取 openId     │
                │    系统不知道用户手机号       │
                └─────────────────────────────┘
                              │
                              ▼
                ┌─────────────────────────────┐

                ┌─────────────────────────────┐
                │  前端弹出引导，提示用户       │
                │  "请授权手机号以完成绑定"     │
                │  button getPhoneNumber       │
                └─────────────────────────────┘

                   ┌──────────┴──────────┐
                   ▼                     ▼
            ┌────────────┐         ┌────────────────┐
            │ 用户授权    │         │ 用户拒绝授权    │
            └──────┬─────┘         └───────┬────────┘
                   │                       │
                   │                       ▼
                   │             ┌────────────────────┐
                   │             │ 降级为普通客户      │
                   │             │ 进入 pages/index   │
                   │             └────────────────────┘
                   ▼
                ┌─────────────────────────────┐
                │  调用 bindPhoneAndRole 接口  │

                ┌─────────────────────────────┐

                │  使用 session_key 解密得到   │
                │  明文手机号                  │
                └─────────────────────────────┘
                              │
                              ▼
                ┌─────────────────────────────┐
                │  匹配 staff_invites 手机号    │
                │  匹配成功 → 创建 portal_roles │
                │  更新 staff_invites:         │
                │    - boundOpenId = openId    │
                │    - status = 2 已绑定       │
                └─────────────────────────────┘
                              │
                              ▼
                ┌─────────────────────────────┐
                │  角色绑定完成                 │
                │  进入对应工作台               │
                │  (admin / store_owner)       │
                └─────────────────────────────┘
```

**技术实现要点：**

1. **微信 openId 与手机号的隔离：** 微信小程序默认仅能获取 `openId`，无法直接获取用户手机号。因此邀请绑定流程必须通过微信官方 `getPhoneNumber` 按钮授权获取加密手机号数据。

2. **绑定流程三步走：**
   - **第一步：** `resolveLaunchContext` 查询 `staff_invites` 表，若存在 `status=1`（待绑定）且 `boundOpenId` 为空的记录，返回 `needsPhoneBind: true` 给前端
   - **第二步：** 前端展示引导页面，用户点击 `<button open-type="getPhoneNumber">` 授权，微信回调返回加密数据 `{ encryptedData, iv }`
   - **第三步：** 调用 `bindPhoneAndRole` 云函数，云函数通过 `cloud.getPhoneNumber({ encryptedData, iv })` 解密得到明文手机号，与 `staff_invites` 表中记录匹配，创建 `portal_roles` 记录

3. **安全性保障：**
   - 手机号加密数据由微信 `session_key` 加密，仅在云函数端可解密
   - 解密后的手机号必须与 `staff_invites` 中超级管理员预填的手机号完全匹配，防止越权绑定
   - 绑定后 `staff_invites.boundOpenId` 写入当前 openId，防止重复绑定

**关键接口：**
- `resolveLaunchContext`：检测待绑定邀请
- `bindPhoneAndRole`：手机号解密 + 角色绑定

**关键数据表：** `staff_invites`、`portal_roles`

### 4.2 代金券状态机

代金券状态机描述代金券从生成到核销（或过期）的状态流转。代金券由世纪恒通调用智机惠 `/order/create` 接口（mobileIntegration 云函数）时生成，生成即激活。

**券状态流转：**

```
   ┌──────────────────────────────────────────────────────────────┐
   │                                                              │
   │  世纪恒通调用 /order/create 生成代金券                          │
   │  （生成即激活，当月有效，至当月最后一天 23:59:59）               │
   │                                                              │
   └──────────────────────────┬───────────────────────────────────┘
                              │
                              ▼
                     ┌────────────────┐
                     │ 1 可使用        │
                     │ (status=1)     │
                     └───────┬────────┘
                             │
            ┌────────────────┼────────────────┐
            │                │                │
            │ 用户主动核销   │ 自动结算        │ 当月过期
            │ (门店核销)     │ (懒触发)        │ (定时/查询触发)
            ▼                ▼                ▼
   ┌────────────────┐ ┌────────────────┐ ┌────────────────┐
   │ 2 已核销        │ │ 券仍为 1        │ │ 3 已过期        │
   │ (status=2)     │ │ (status=1)     │ │ (status=3)     │
   │ settleType=    │ │ settleType=    │ │ （终态）        │
   │   manual       │ │   auto         │ │                │
   │ 回传世纪恒通   │ │ 回传世纪恒通   │ │                │
   │ （终态，不可逆）│ │ 用户仍可使用   │ │                │
   └────────────────┘ └────────────────┘ └────────────────┘
```

**settleType 结算类型说明：**

| settleType | 含义 | 触发方式 | 券状态变化 | 用户可见 |
|:----------:|------|----------|:----------:|:--------:|
| `manual` | 用户主动核销 | 门店扫码核销 | status=1→2 | 券显示"已核销" |
| `auto` | 自动结算 | 懒触发（到期系统自动回传，CAS 并发保护） | status 保持=1 | 券仍显示"可使用" |

**状态定义：**

| 状态值 | 状态名 | 说明 | 触发动作 |
|:------:|--------|------|----------|
| 1 | 可使用 | 世纪恒通调用 `/order/create` 生成，立即激活，当月有效（至当月最后一天 23:59:59） | 世纪恒通调用 `/order/create` |
| 2 | 已核销 | 门店已完成核销，settleType='manual' | `storeConfirmVerify`（需传 `verifyCode`） |
| 3 | 已过期 | 超过当月有效期 | 定时/查询触发自动变更 |

**关键规则：**
- 券状态仅有 1（可使用）、2（已核销）、3（已过期），无 `status=0`（待激活），券生成即激活（status=1）
- 当月有效，有效期至当月最后一天 23:59:59
- `status=2` 和 `status=3` 均为终态，不可逆
- `settleType='auto'` 时券状态仍为 `status=1`，代金券在自动结算后仍可继续使用，仅后台记录结算类型
- 自动结算通过懒触发实现（用户查询卡包/生成核销码时检查），使用 CAS 条件更新进行并发保护，调用 `notifyHengtongSettle` 向世纪恒通上报 `verifyStatus=1`，不影响前端展示

| `contracts` | 合约记录 | phone, storeId, status |
| `coupons` | 代金券记录 | phone, storeId, status, settleType, settleAt, createTime, expireTime |

## 6. 异常处理与边界说明

### 6.1 登录与角色分流异常

| 异常场景 | 处理策略 |
|----------|----------|
| `login` 云函数失败 | 提示网络异常，提供重试按钮 |
| `resolveLaunchContext` 超时 | 降级为普通客户身份进入首页 |
| 用户拒绝手机号授权 | 降级为普通客户，不绑定角色 |
| 邀请记录手机号与授权手机号不匹配 | 提示"手机号与邀请不匹配"，不执行绑定 |

### 6.2 合约办理异常

| 异常场景 | 处理策略 |
|----------|----------|
| 验证码被驳回（status=8） | 允许客户重新输入，回到 status=3 |
| 管理员核验拒绝（status=0） | 客户可重新提交，回到 status=1 |
| 收货信息填写超时 | 保留 status=2，允许继续填写 |
| 合约办理中服务异常 | 管理员可手动回退至 status=3 |
| status=5 自动注册失败（bindPhoneToUser） | 记录错误日志，合约状态不变，人工介入补注册 |

### 6.3 代金券与核销异常

| 异常场景 | 处理策略 |
|----------|----------|
| 世纪恒通 `/order/create` 请求签名校验失败（3DES/MD5） | 拒绝生成，记录日志告警 |
| `/order/create` 传入 phone 在 contracts 表无匹配 | 仍生成代金券，storeId 为空（通用券） |
| `/order/create` 传入 phone 在 users 表无对应 openId | 仍生成代金券，用户下次登录绑定手机号后可通过 phone 查询到 |
| 核销码过期（超过 3 分钟） | 提示用户重新生成核销码 |
| `storeConfirmVerify` 缺少 verifyCode 参数 | 拒绝核销，提示"缺少核销码校验参数" |
| 重复核销（券已为 status=2） | 拒绝核销，提示"券已核销" |
| 券已过期（status=3） | 拒绝核销，提示"券已过期" |
| 门店归属校验失败（门店券 storeId 不匹配） | 拒绝核销，提示"非本门店代金券" |
| 自动结算懒触发时 `notifyHengtongSettle` 回传世纪恒通失败 | 记录错误日志，下次查询时重试 |
| 自动结算 CAS 并发更新失败 | 记录日志，下次查询时重试，避免重复结算 |
| 用户无手机号（未授权 getPhoneNumber）查询卡包 | 引导用户手机号授权，绑定后按 phone 查询 |

### 6.4 邀请绑定异常

| 异常场景 | 处理策略 |
|----------|----------|
| 邀请已过期 | 提示"邀请已失效"，联系超级管理员重新发起 |
| 邀请已被绑定（boundOpenId 已存在） | 提示"邀请已被使用"，拒绝重复绑定 |
| 手机号解密失败（cloud.getPhoneNumber 异常） | 提示重新授权，记录错误日志 |
