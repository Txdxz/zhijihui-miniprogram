## 2. 项目背景与业务概述

### 2.1 业务背景

智机惠是一个通过世纪恒通与中国移动权益优选对接的**宠物门店服务代金券核销小程序**。整体业务闭环如下：

1. 用户在中国移动办理业务（如办理套餐、宽带等）后获得权益资格
2. 用户在智机惠小程序提交手机号、选择门店、填写收货信息、回填验证码，完成合约办理
3. 合约办理完成后，系统自动将客户手机号绑定到 openId，完成用户注册（不再预生成代金券）
4. 用户自行前往中国移动权益优选小程序点击"领取"
5. 世纪恒通作为接入方调用智机惠 `mobileIntegration` 云函数的 `/order/create` 接口创建代金券（携带 phone 等参数，按手机号匹配门店：有合约→门店券，无合约→通用券），代金券当月有效
6. 用户到智机惠合作的宠物门店出示核销码
7. 门店店员扫码核销，核销状态通过随机时间自动结算回传世纪恒通（verifyStatus=1），代金券在自动结算后仍可正常使用

### 2.2 业务价值

- **对中国移动**：为移动用户提供可感知的实物权益，提升业务办理转化率
- **对宠物门店**：获取中国移动导流的客户，提升门店客流与营收
- **对用户**：办理移动业务即可获得宠物门店代金券，到店核销使用

### 2.3 业务范围

| 范围 | 说明 |
|------|------|
| 客户端 | 微信小程序，面向 C 端用户办理合约、查看代金券、生成核销码 |
| 门店端 | 同一小程序内的门店核销工作台，面向店员/门店负责人 |
| 管理后台 | 同一小程序内的管理分包，面向管理员/超级管理员 |
| 对接方 | 世纪恒通科技股份有限公司（接入方），承接中国移动各省分公司权益优选小程序的权益发放与核销回传 |

### 3.2 架构特点

| 特点 | 说明 |
|------|------|
| **全云开发** | 无独立服务器，前端、云函数、数据库、存储均在微信云开发内，运维成本极低 |
| **Serverless** | 云函数按需执行，自动扩缩容，无需关心服务器资源 |
| **单一小程序多端** | 客户端、门店端、管理后台集成在同一小程序内，通过角色分流 |
| **NoSQL 数据库** | 云数据库为文档型数据库（类 MongoDB），schema-free，便于迭代 |
| **前后端同源** | 前端与云函数在同一项目内，便于联调与版本管理 |
| **安全收敛** | 所有数据库集合 `read:false, write:false`，只能通过云函数访问 |

## 4. 技术选型说明

### 4.1 前端技术栈

| 技术 | 版本/说明 | 用途 |
|------|----------|------|
| 微信小程序原生开发 | 基础库 3.15.1+ | 不依赖 Taro/Uni-app 等框架，直接使用微信原生语法 |
| JavaScript (ES6+) | 开发者工具开启 ES6 转译 | 业务逻辑编写 |
| WXML / WXSS | 微信原生 | 视图与样式 |
| Vant Weapp | @vant/weapp ^1.11.7 | UI 组件库（弹窗、表单等） |
| weapp-qrcode | ^1.0.0 | 核销码二维码生成 |
| @wxobs/miniprogram-helper | ^0.6.10 | 微信官方数据分析 SDK 埋点 |

**选型理由：**

- **原生开发而非框架**：项目业务逻辑集中在合约办理与核销流程，原生开发性能最优、包体积最小，避免框架运行时开销
- **Vant Weapp**：成熟的小程序 UI 组件库，提供丰富的移动端组件，减少重复造轮子
- **weapp-qrcode**：核销码需要在前端生成二维码图片，纯前端生成避免云函数调用开销

### 4.2 后端技术栈

| 技术 | 用途 |
|------|------|
| 微信云开发（CloudBase） | 全托管后端服务 |
| wx-server-sdk | 云函数 Node.js SDK，提供数据库、存储、云调用能力 |
| 云函数（Node.js） | 业务逻辑处理、权限校验、数据操作 |
| 云数据库（文档型 NoSQL） | 数据持久化，类 MongoDB 语法 |
| 云存储 | 图片等静态资源存储 |

**选型理由：**

- **无独立服务器**：项目体量与团队规模适合全云开发方案，省去服务器运维、域名备案、HTTPS 证书等成本
- **云函数自动扩缩容**：应对活动高峰无需提前扩容
- **wx-server-sdk 原生集成**：可直接获取 openId、调用 getPhoneNumber 等微信能力，无需额外鉴权

### 4.3 不采用独立后端的原因

1. **运维成本**：项目为面向中国移动各省分公司的业务，用户量可控，无需独立后端的弹性能力
2. **开发效率**：前后端同项目，联调成本低，版本一致性好
3. **微信能力集成**：getPhoneNumber、小程序码等微信能力在云函数内调用最便捷
4. **成本控制**：云开发按量计费，低峰期成本极低

## 6. 角色与权限体系

### 6.1 角色定义

系统定义 5 种角色，存储在 `portal_roles` 集合中：

| 角色键 | 角色名称 | 权限范围 | 典型权限 |
|--------|---------|---------|---------|
| `customer` | 客户 | 全局 | 查看本人合约、查看本人代金券、生成核销码 |
| `store_clerk` | 店员 | 单门店 | 查询核销码、确认核销、查看本店核销记录 |
| `store_owner` | 门店负责人 | 单门店 | 门店核销、查看本店核销记录（不管理门店，门店CRUD由 super_admin 负责） |
| `admin` | 管理员 | 全局 | 合约管理、门店列表查看、运营配置 |
| `super_admin` | 超级管理员 | 全局 | 全部权限，包括门店增删改查、代金券规则管理、管理员账号管理、角色分配与回收 |

### 6.2 角色模型设计

```
┌─────────────────────────────────────────────────────┐
│                  portal_roles 集合                   │
├─────────────────────────────────────────────────────┤
│  openId      : 微信用户 openId                       │
│  roleKey     : 角色键（customer/store_clerk/...）    │
│  scopeType   : 权限范围类型（global/store/system）   │
│  scopeId     : 权限范围 ID（门店 ID，全局角色为空）  │
│  scopeName   : 范围名称（如门店名）                  │
│  name        : 角色显示名称                          │
│  phone       : 绑定手机号                            │
│  permissions : 权限列表（如 ['coupon.verify']）      │
│  status      : 状态（1=有效, 0=已禁用）              │
│  boundBy     : 绑定方式（auto_bind/self_bind/...）   │
│  createdAt   : 创建时间                              │
│  updatedAt   : 更新时间                              │
└─────────────────────────────────────────────────────┘
```

**关键设计：**

- **一个用户可有多个角色**：例如同一 openId 可以既是客户又是某门店负责人
- **scopeType + scopeId 精确控制范围**：门店角色绑定具体 storeId，管理员为全局
- **status 软删除**：禁用角色时设为 0，保留历史记录

### 6.3 角色识别与分流流程

```
[小程序启动]
    │
    ▼
[app.js onLaunch] ──初始化云开发──▶ [读取本地缓存的角色]
    │
    ▼
[pages/launch/index 启动页]
    │
    ▼
[调用 resolveLaunchContext 云函数]
    │
    ▼
[云函数查询 portal_roles where openId = 当前用户]
    │
    ├── 无角色 ──▶ 检查 staff_invites 待绑定邀请
    │               ├── 有邀请且手机号匹配 ──▶ 自动绑定 ──▶ 返回角色入口
    │               ├── 有邀请但需绑定 ──▶ 引导到 /pages/bind/index
    │               └── 无邀请 ──▶ 返回 customer 入口
    │
    ├── 单角色 ──▶ 直接返回该角色入口
    │
    └── 多角色 ──▶ 根据 lastPortalRole 记忆 / 返回角色选择列表
                    │
                    ▼
            [前端跳转到对应页面]
            ├── customer     ──▶ /pages/index/index
            ├── store_owner  ──▶ /pages/store/verify?storeId=xxx
            ├── store_clerk  ──▶ /pages/store/verify?storeId=xxx
            ├── admin        ──▶ /packageAdmin/pages/admin/index
            └── super_admin  ──▶ /packageAdmin/pages/admin/index
```

### 6.4 权限校验机制

权限校验**全部在云函数内执行**，前端不做任何权限判断（前端仅做界面展示控制）。云函数提供三个核心校验函数：

| 函数 | 位置 | 用途 |
|------|------|------|
| `ensureRole(openId, roleKeys)` | `utils/permission.js` + 各云函数内 | 校验用户是否具有指定角色之一 |
| `ensureSuperAdmin(openId)` | 同上 | 校验用户是否为超级管理员 |
| `ensureStoreRole(openId, storeId)` | 同上 | 校验用户是否具有指定门店的角色 |

**调用示例：**

```javascript
// 云函数内权限校验
async function handleAdminUpdateStatus(openId, event) {
  const isAdmin = await ensureRole(openId, ['admin', 'super_admin']);
  if (!isAdmin) {
    return { code: 403, message: '当前账号无管理员权限' };
  }
  // ... 业务逻辑
}
```

### 7.2 合约办理模块

**职责：** 提交手机号 → 资格核验 → 填写收货信息 → 验证码验证 → 合约完成 → 发货签收

**业务流程：**

```
[用户选择门店 + 输入手机号]
    │
    ▼ submitPhone
[创建合约记录，status=1 待核验]
    │

    │

    │
    ▼ 管理员确认办理完成（adminUpdateStatus）
[status=5 合约已办] ──▶ 调用bindPhoneToUser自动注册（仅绑定手机号，不再生成代金券）
```

- **currentContractId 指针**：`portal_users.currentContractId` 记录用户当前合约，避免每次查询

### 7.3 权益管理模块

**职责：** 世纪恒通 `/order/create` 驱动生成代金券 → 按手机号匹配门店 → 当月有效 → 到店核销 → 状态回传世纪恒通

**代金券生命周期：**

```
[用户在中国移动权益优选小程序点击"领取"]
    │
    ▼ 世纪恒通调用 mobileIntegration /order/create
[接收 orderNo, rechargeAccount(phone), skuId, notifyUrl, expandData]
    │
    ▼ 按手机号匹配门店
    ├── 有合约记录 ──▶ 生成门店券（绑定具体门店）
    └── 无合约记录 ──▶ 生成通用券（全门店通用）
    │
    ▼ 创建代金券，status=1 可使用，计算当月有效期（当月最后一天 23:59:59）
    │
    ▼ 用户在卡包生成核销码 generateVerifyCode()
[核销码有效期3分钟]
    │
    ├── 当月到期未使用 ──▶ status=3 已过期
    │
    └── 到店核销 ──▶ status=2 已核销 ──▶ 回传世纪恒通核销状态（verifyStatus=1）
```

**关键设计：**

- **世纪恒通 `/order/create` 驱动生成券**：代金券不再由合约完成时预生成，也不再由 `portalBiz` 的 `activateCoupon` 生成（该 action 已删除），而是由世纪恒通调用 `mobileIntegration` 云函数的 `/order/create` 接口实时创建。用户在中国移动权益优选小程序点击"领取"后，世纪恒通携带 `orderNo`、`rechargeAccount`（即手机号）、`skuId`、`notifyUrl`、`expandData` 等参数触发智机惠创建代金券
- **按手机号匹配门店**：根据 `rechargeAccount`（手机号）查询用户最近的合约记录，有合约则生成绑定该门店的门店券，无合约则生成通用券（全门店可用）
- **代金券规则匹配**：根据 `coupon_rules` 集合的 `storeScope`（all/bound）和 `selectedStores` 匹配面额
- **当月有效期**：从世纪恒通创建代金券时开始计算，有效期至当月最后一天 23:59:59
- **核销码3分钟有效**：防止核销码被截图盗用
- **世纪恒通对接字段**：`hengtongOrderNo`、`notifyUrl`、`expandData`、`skuId`、`cardNo` 用于与世纪恒通对账；兼容字段 `mobileBenefitId`、`mobileOrderId`、`mobileNotifiedAt` 保留用于历史数据
- **客服电话**：卡包页面底部展示客服电话 4007071506（服务时间 09:00-17:30，周一至周五），用户可点击拨打

**关键文件：**

- `cloudfunctions/mobileIntegration/index.js`（`handleCreateOrder`、`handleQueryOrder`、`notifyHengtongSettle`）
- `cloudfunctions/mobileIntegration/config.js`（appId、appSecret、desKey 配置）
- `cloudfunctions/portalBiz/index.js`（`handleGenerateVerifyCode`、`tryAutoSettle`、`notifyMobileSettle`）
- `pages/coupon/coupon.js`（卡包页面，含客服电话）

### 7.4 门店核销模块

**职责：** 扫码核销 → 门店权限校验 → 核销记录（store_clerk 和 store_owner 均可操作）

**核销流程：**

```
[客户出示核销码（6位数字）]
    │
    ▼ 门店店员输入/扫码
[storeVerifyCoupon(verifyCode, storeId)]
    │
    ▼ 云函数校验
    ├── ensureStoreRole(openId, storeId) 门店权限校验
    ├── 校验核销码是否存在
    ├── 校验核销码是否属于当前门店
    ├── 校验核销码是否已使用
    └── 校验核销码是否过期（3分钟）
    │
    ▼ 返回代金券预览信息
[店员确认核销]
    │
    ▼ storeConfirmVerify(couponId, verifyCode)
    │
    ▼ 云函数执行核销（CAS 条件更新，并发保护）
    ├── 校验 verifyCode 与券内 verifyCode 一致
    ├── 条件更新：仅当 status=1 时才核销（where _id + status=1）
    ├── 代金券 status=1 → status=2
    ├── 记录 usedAt 核销时间
    ├── 清空 verifyCode
    ├── 设置 settleType='manual'
    └── 异步回传世纪恒通核销状态（notifyMobileSettle → notifyHengtongSettle，verifyStatus=1）
    │
    ▼ 返回核销成功
```

**关键设计：**

- **storeConfirmVerify 需 verifyCode 参数**：确认核销时除 `couponId` 外必须传 `verifyCode`，云函数会校验核销码与券内 `verifyCode` 是否一致，防止越权核销
- **CAS 并发保护**：核销使用条件更新 `where({ _id, status: 1 })`，仅当券状态为可使用时才更新为已核销，避免并发请求重复核销
- **自动结算也使用 CAS**：`tryAutoSettle` 使用 `where({ _id, settleType: _.or(_.eq(''), _.exists(false)) })` 条件更新，仅当未结算时才标记为 auto 结算，避免并发重复回传

**核销记录查询：**

- `handleGetStoreVerifyRecords`：按月份分组返回门店核销记录，含今日核销总额
- 手机号脱敏展示（`maskPhone` 函数）

**关键文件：**

- `cloudfunctions/portalBiz/index.js`（`handleStoreVerifyCoupon`、`handleStoreConfirmVerify`、`handleGetStoreVerifyRecords`、`tryAutoSettle`、`notifyMobileSettle`）
- `pages/store/verify.js`（门店核销工作台）

### 8.3 云函数统一规范

所有云函数遵循以下规范：

1. **统一返回格式**：`{ code: number, data?: any, message?: string }`
   - `code: 200` 成功
   - `code: 400` 参数错误
   - `code: 403` 权限不足
   - `code: 404` 资源不存在
   - `code: 429` 频率限制
   - `code: 500` 服务异常

2. **统一异常捕获**：`main` 函数外层 try-catch，避免云函数直接抛错

3. **统一初始化**：`cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })` 使用动态环境变量

4. **统一鉴权**：通过 `cloud.getWXContext()` 获取 OPENID，不信任前端传入的 openId

5. **统一频率限制**：调用 `checkRateLimit(OPENID, action)` 进行限流

## 10. 与世纪恒通对接设计

### 10.1 对接架构

智机惠通过**世纪恒通科技股份有限公司**（以下简称"世纪恒通"）作为第三方权益接入平台，承接中国移动权益优选小程序的权益发放。对接角色定义：

- **接入方**：世纪恒通（调用智机惠接口）
- **服务提供方**：智机惠（提供代金券创建、查询、核销回传接口）

```
┌─────────────────┐                    ┌─────────────────────┐
│   智机惠小程序   │                    │  世纪恒通           │
│ （服务提供方）  │                    │  （接入方）         │
│                 │                    │                     │
│  1.用户办合约   │                    │                     │
│  （仅绑定手机   │                    │                     │
│   号，不生成券）│                    │                     │
│                 │                    │                     │
│                 │                    │  2.用户在中国移动   │
│                 │                    │  权益优选小程序     │
│                 │                    │  点击"领取"         │
│                 │                    │                     │
│                 │  ◀── 3./order/create ─│ 世纪恒通调用      │
│  4.按手机号匹   │                    │  携带 orderNo、     │
│  配门店生成券   │                    │  rechargeAccount、  │
│  status=1可使用 │                    │  notifyUrl 等       │
│  当月有效       │                    │                     │
│                 │                    │                     │
│  5.用户到店核销 │                    │                     │
│                 │                    │                     │
│                 │  ── 6.notifyUrl 回调 ─▶│ 智机惠回传       │
│                 │                    │  verifyStatus=1     │
│                 │                    │  （随机1-25天延迟） │
│                 │                    │                     │
└─────────────────┘                    └─────────────────┘
```

**关键设计说明：**

- **代金券由世纪恒通 `/order/create` 生成**：代金券不再由合约完成时预生成，也不再由 `portalBiz` 的 `activateCoupon` 生成（该 action 已删除），而是由世纪恒通调用 `mobileIntegration` 云函数的 `/order/create` 接口实时创建。用户在中国移动权益优选小程序点击"领取"后，世纪恒通携带 `orderNo`、`rechargeAccount`（手机号）、`skuId`、`notifyUrl`、`expandData` 等参数触发智机惠创建代金券
- **按手机号匹配门店**：根据 `rechargeAccount`（手机号）查询用户最近的合约记录，有合约则生成绑定该门店的门店券，无合约则生成通用券（全门店可用）
- **当月有效**：代金券有效期至当月最后一天 23:59:59
- **用户自行前往领取**：用户需自行打开中国移动权益优选小程序点击"领取"，智机惠不提供跳转链接

### 10.2 鉴权与加密

**鉴权方式：** appId + appSecret + MD5 签名（32 位小写）

**签名算法：** `MD5(appId + traceId + timestamp + nonceStr + 业务参数字典排序拼接 + appSecret)`

- 业务参数按 key 字典排序，拼接为 `key1value1key2value2...`（跳过空值）
- 签名结果以 32 位小写 hex 输出
- 请求头传递：`appId`、`traceId`、`timestamp`、`nonceStr`、`sign`

**加密方式：** 3DES-ECB/PKCS5Padding，输出 hex

- 请求体与响应体均使用 `ciphertext` 字段包裹密文
- 密钥由世纪恒通提供，配置在 `cloudfunctions/mobileIntegration/config.js` 的 `desKey`
- 未配置 `desKey` 时明文传输（仅开发阶段）

**配置文件：** `cloudfunctions/mobileIntegration/config.js`

```javascript
module.exports = {
  appId: 'zjh_appid_001',        // 智机惠提供，分享给世纪恒通
  appSecret: 'zjh_secret_2026',  // 智机惠提供，分享给世纪恒通
  desKey: '',                    // 世纪恒通提供，待替换
  hengtongApiBase: '',           // 世纪恒通API域名（回调用），待替换
};
```

### 10.3 对接接口

所有接口由 `mobileIntegration` 云函数承载，通过 HTTP 触发，路径前缀即接口名。

#### 10.3.1 创建权益充值订单（世纪恒通 → 智机惠）

**接口：** `POST /order/create`

**调用方：** 世纪恒通（接入方）

**鉴权：** appId + appSecret + MD5 签名

**请求参数（解密后业务参数）：**

```javascript
{
  orderNo: 'HT20260624XXXX',       // 世纪恒通订单号（幂等主键）
  rechargeAccount: '13800138000',  // 用户手机号（用于匹配门店）
  skuId: 'SKU_001',                // 商品 SKU ID
  notifyUrl: 'https://xxx/notify', // 核销状态回调地址
  expandData: '...',               // 透传扩展数据
  number: 1                        // 数量
}
```

**处理逻辑：**

1. 验证签名（`verifySign`）
2. 幂等检查：按 `orderNo`（即 `hengtongOrderNo`）查重，已存在则直接返回已有券信息
3. 根据 `rechargeAccount`（手机号）查询用户最近合约记录（status >= 5）
4. 匹配门店：有合约 → 生成门店券（绑定具体门店），无合约 → 生成通用券（全门店通用）
5. 匹配代金券规则（面额等）
6. 创建代金券记录，status=1（可使用），计算当月有效期（当月最后一天 23:59:59）
7. 计算随机结算时间（1-25 天内的随机白天时间，写入 `settleAt`）
8. 返回创建结果

**响应（加密前）：**

```javascript
{
  code: 200,
  msg: '成功',
  data: {
    orderNo: 'HT20260624XXXX',
    spOrderNo: 'CP20260624XXXX',   // 智机惠代金券 ID
    rechargeStatus: 1,
    cards: [{
      cardNo: 'CP20260624XXXX',
      cardPassword: '',
      cardDeadline: '2026-06-30 23:59:59'  // 当月最后一天
    }]
  }
}
```

**调用方：** 世纪恒通（接入方）

#### 10.3.3 核销状态回调（智机惠 → 世纪恒通）

**接口：** `POST {notifyUrl}`（由世纪恒通在 `/order/create` 时传入）

**调用方：** 智机惠（服务提供方）

**鉴权：** 智机惠使用相同的 appId + appSecret + MD5 签名方案对请求签名

**加密：** 请求体使用 3DES-ECB 加密，包裹在 `ciphertext` 字段

**触发时机：**

- **自动结算**：代金券创建时即设定随机结算时间（1-25 天），在用户查询券或生成核销码时由 `tryAutoSettle` 懒触发检查，到达结算时间且未结算则自动回传 `verifyStatus=1`。**代金券在自动结算后仍可正常使用**，`settleType='auto'` 仅记录已回传状态
- **手动核销**：门店确认核销（`storeConfirmVerify`）后，设置 `settleType='manual'`，异步回传 `verifyStatus=1`

**并发保护（CAS）：**

- 自动结算使用条件更新 `where({ _id, settleType: _.or(_.eq(''), _.exists(false)) })`，仅当未结算时才标记为 auto，避免并发重复回传
- 手动核销使用条件更新 `where({ _id, status: 1 })`，仅当券可使用时才核销

**回传内容（加密前业务参数）：**

```javascript
{
  orderNo: 'HT20260624XXXX',
  spOrderNo: 'CP20260624XXXX',
  rechargeAccount: '13800138000',
  rechargeStatus: 1,
  orderCreateTime: '2026-06-24 10:00:00',
  rechargeCompleteTime: '2026-06-24 10:00:00',
  expandData: '...',
  cards: [{
    cardNo: 'CP20260624XXXX',
    cardPassword: '',
    cardDeadline: '2026-06-30 23:59:59',
    verifyStatus: 1,            // 固定为 1（已核销）
    verifyTime: '2026-06-25 14:30:00'
  }]
}
```

**实现位置：** `cloudfunctions/mobileIntegration/index.js` 的 `notifyHengtongSettle` 函数，由 `portalBiz` 的 `notifyMobileSettle` 通过 `require('./mobileIntegration/index')` 调用。

### 10.4 对接注意事项

1. **幂等性**：`/order/create` 接口按 `orderNo`（`hengtongOrderNo`）幂等，同一订单号多次调用只创建一次（通过查询 `hengtongOrderNo` 是否已存在实现）
2. **手机号作为查询主键**：`rechargeAccount`（手机号）字段是世纪恒通调用时匹配门店和用户的核心查询字段，`coupons` 集合需建立 `phone` 索引
3. **自动结算容错**：核销状态回传失败不阻塞用户使用，仅记录日志，后续可通过懒触发再次尝试
4. **密钥保护**：`appId`、`appSecret`、`desKey` 当前配置在 `config.js` 中，建议迁移到云环境变量
5. **自动注册机制**：合约办理完成（status=5）时，系统自动调用 `bindPhoneToUser` 将客户手机号绑定到 openId，完成用户注册。这意味着用户首次办理合约时即自动成为注册用户，无需额外注册步骤
6. **代金券生成入口唯一**：旧的 `portalBiz` 的 `activateCoupon` action 已删除，代金券生成统一由 `mobileIntegration` 的 `/order/create` 承接，避免双入口
7. **HTTP 触发配置**：`mobileIntegration` 云函数需在云开发控制台配置 HTTP 触发路径，世纪恒通通过 HTTPS 调用

### 12.2 数据库安全

**所有数据库集合权限设置为：**

```json
{
  "read": false,
  "write": false
}
```

**含义：**

- 客户端（小程序前端）无法通过 `wx.cloud.database()` 直接读写任何集合
- 所有数据操作必须通过云函数 `wx.cloud.callFunction` 间接访问
- 云函数内通过 `cloud.database()` 拥有完整权限

**配置位置：** 微信云开发控制台 → 数据库 → 对应集合 → 权限设置 → 自定义安全规则

#### 12.3.2 世纪恒通接口签名鉴权

世纪恒通调用智机惠 `mobileIntegration` 云函数的 `/order/create`、`/order/query` 接口时，采用 appId + appSecret + MD5 签名鉴权：

```javascript
// 签名生成（mobileIntegration/index.js）
function generateSign(params, appId, traceId, timestamp, nonceStr, appSecret) {
  const sortedKeys = Object.keys(params).sort();
  let paramStr = '';
  for (const key of sortedKeys) {
    const val = params[key];
    if (val !== '' && val !== null && val !== undefined) {
      paramStr += key + val;
    }
  }
  const signStr = appId + traceId + timestamp + nonceStr + paramStr + appSecret;
  return crypto.createHash('md5').update(signStr, 'utf8').digest('hex');
}

// 签名验证
function verifySign(params, headers) {
  const { appId, traceId, timestamp, nonceStr, sign } = headers;
  if (!appId || !traceId || !timestamp || !nonceStr || !sign) return false;
  if (appId !== config.appId) return false;
  const expectedSign = generateSign(params, appId, traceId, timestamp, nonceStr, config.appSecret);
  return expectedSign === sign;
}
```

**请求头鉴权字段：** `appId`、`traceId`、`timestamp`、`nonceStr`、`sign`

**请求体加密：** 业务参数使用 3DES-ECB/PKCS5Padding 加密，输出 hex，包裹在 `ciphertext` 字段中传输

> **⚠️ 安全提示：** 当前 `appId`、`appSecret`、`desKey` 配置在 `cloudfunctions/mobileIntegration/config.js` 中，正式上线前应将 `desKey`（世纪恒通提供）填入，并建议将所有密钥迁移到云环境变量。世纪恒通接口通过 `rechargeAccount`（手机号）字段匹配门店，不再依赖 `contractId`。

### 12.4 接口频率限制

基于 `rate_limits` 集合实现，按 `openId + action` 维度限流，1分钟滑动窗口：

| 操作类型 | 频率限制 | 说明 |
|---------|---------|------|
| 查询类（getStores, getCoupons 等） | 120/分钟 | 宽松，允许列表翻页 |
| 默认 | 60/分钟 | 兜底 |
| 写入类（submitPhone, submitOrderInfo 等） | 10/分钟 | 防止刷单 |
| 敏感操作（createNewContract, adminCreateStore） | 5/分钟 | 防止恶意创建 |
| 删除操作（adminDeleteStore, deleteCouponRule） | 3/分钟 | 防止误操作 |
| 身份绑定（bindRole） | 5/分钟 | 防止暴力绑定 |
| 短信发送（sendSmsCode） | 3/分钟 | 防止短信轰炸 |

**实现位置：** `cloudfunctions/rateLimit.js`（各云函数通过 `require('./rateLimit')` 引入）

### 12.5 参数验证

云函数内通过 `validateParams` 函数统一校验参数：

```javascript
function validateParams(params = {}, rules = {}) {
  // 支持：required(必填)、type(类型)、min/max(数值范围)
  //       minLength/maxLength(字符串长度)、pattern(正则)、message(错误提示)
}
```

**典型校验规则：**

- 手机号：`/^1[3-9]\d{9}$/`
- 验证码：`/^\d{6}$/`
- 必填字段：`required: true`
- 数值范围：`min: 0, max: 100`

云函数内 `sanitizeForLog` 函数对敏感信息脱敏后再记录日志：

| 类型 | 脱敏规则 | 示例 |
|------|---------|------|

| name | 首字+* | `张**` |

### 12.7 安全注意事项

1. **initDatabase 云函数必须删除**：上线前在云开发控制台删除 `initDatabase` 云函数，避免被恶意调用初始化数据
2. **世纪恒通密钥保护**：`mobileIntegration/config.js` 中的 `appId`、`appSecret`、`desKey` 为对接密钥，`desKey` 待世纪恒通提供后填入，建议所有密钥迁移到云环境变量
3. **企业微信 Webhook 密钥保护**：`portalBiz/index.js` 中的 `WECOM_WEBHOOK` 含群机器人 key，避免泄露
4. **超级管理员 openId 保护**：`portal_roles` 中 super_admin 记录的 openId 是系统最高权限，避免泄露
5. **云函数日志查看权限**：云开发控制台日志可能含敏感信息，需控制访问权限
6. **mobileIntegration HTTP 触发路径保护**：`mobileIntegration` 云函数通过 HTTP 触发，需确保仅世纪恒通可调用（依赖签名鉴权），避免被恶意调用伪造代金券
