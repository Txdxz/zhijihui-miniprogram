## 2. 项目背景与业务概述

### 2.1 业务背景

智机惠是一个与中国移动权益平台对接的**宠物门店服务代金券核销小程序**。整体业务闭环如下：

1. 用户在中国移动办理业务（如办理套餐、宽带等）后获得权益资格
2. 用户在智机惠小程序提交手机号、选择门店、填写收货信息、回填验证码，完成合约办理
3. 合约办理完成后，智机惠为用户预生成代金券（待激活状态）
4. 用户前往中国移动【河北移动权益优选】小程序领取权益
5. 移动平台通过 API 回调通知智机惠激活代金券（状态由"待激活"变为"可使用"）
6. 用户到智机惠合作的宠物门店出示核销码
7. 门店店员扫码核销，核销状态回传中国移动

### 2.2 业务价值

- **对中国移动**：为移动用户提供可感知的实物权益，提升业务办理转化率
- **对宠物门店**：获取中国移动导流的客户，提升门店客流与营收
- **对用户**：办理移动业务即可获得宠物门店代金券，到店核销使用

### 2.3 业务范围

| 范围 | 说明 |
|------|------|
| 客户端 | 微信小程序，面向 C 端用户办理合约、查看代金券、生成核销码 |
| 门店端 | 同一小程序内的门店核销工作台，面向店员/门店负责人 |
| 管理后台 | 同一小程序内的管理分包，面向管理员/超级管理员 |
| 对接方 | 中国移动河北移动权益优选小程序（权益发放与核销回传） |

## 3. 总体架构设计

### 3.1 架构总览

智机惠采用**微信云开发（CloudBase）全托管架构**，无独立服务器，前端、后端、数据库、存储全部基于微信云开发生态。

### 3.2 架构特点

| 特点 | 说明 |
|------|------|
| **全云开发** | 无独立服务器，前端、云函数、数据库、存储均在微信云开发内，运维成本极低 |
| **Serverless** | 云函数按需执行，自动扩缩容，无需关心服务器资源 |
| **单一小程序多端** | 客户端、门店端、管理后台集成在同一小程序内，通过角色分流 |
| **NoSQL 数据库** | 云数据库为文档型数据库（类 MongoDB），schema-free，便于迭代 |
| **前后端同源** | 前端与云函数在同一项目内，便于联调与版本管理 |
| **安全收敛** | 所有数据库集合 `read:false, write:false`，只能通过云函数访问 |

## 4. 技术选型说明

### 4.1 前端技术栈

| 技术 | 版本/说明 | 用途 |
|------|----------|------|
| 微信小程序原生开发 | 基础库 3.15.1+ | 不依赖 Taro/Uni-app 等框架，直接使用微信原生语法 |
| JavaScript (ES6+) | 开发者工具开启 ES6 转译 | 业务逻辑编写 |
| WXML / WXSS | 微信原生 | 视图与样式 |
| Vant Weapp | @vant/weapp ^1.11.7 | UI 组件库（弹窗、表单等） |
| weapp-qrcode | ^1.0.0 | 核销码二维码生成 |
| @wxobs/miniprogram-helper | ^0.6.10 | 微信官方数据分析 SDK 埋点 |

**选型理由：**

- **原生开发而非框架**：项目业务逻辑集中在合约办理与核销流程，原生开发性能最优、包体积最小，避免框架运行时开销
- **Vant Weapp**：成熟的小程序 UI 组件库，提供丰富的移动端组件，减少重复造轮子
- **weapp-qrcode**：核销码需要在前端生成二维码图片，纯前端生成避免云函数调用开销

### 4.2 后端技术栈

| 技术 | 用途 |
|------|------|
| 微信云开发（CloudBase） | 全托管后端服务 |
| wx-server-sdk | 云函数 Node.js SDK，提供数据库、存储、云调用能力 |
| 云函数（Node.js） | 业务逻辑处理、权限校验、数据操作 |
| 云数据库（文档型 NoSQL） | 数据持久化，类 MongoDB 语法 |
| 云存储 | 图片等静态资源存储 |

**选型理由：**

- **无独立服务器**：项目体量与团队规模适合全云开发方案，省去服务器运维、域名备案、HTTPS 证书等成本
- **云函数自动扩缩容**：应对活动高峰无需提前扩容
- **wx-server-sdk 原生集成**：可直接获取 openId、调用 getPhoneNumber 等微信能力，无需额外鉴权

### 4.3 不采用独立后端的原因

1. **运维成本**：项目为区域性业务（河北移动），用户量可控，无需独立后端的弹性能力
2. **开发效率**：前后端同项目，联调成本低，版本一致性好
3. **微信能力集成**：getPhoneNumber、订阅消息、小程序码等微信能力在云函数内调用最便捷
4. **成本控制**：云开发按量计费，低峰期成本极低

- **主包**：客户端页面（首页、卡包、合约详情、门店核销、绑定、启动页），所有用户都会使用

- **预加载规则**：在启动页和首页预加载管理后台分包，确保管理员切换身份时无延迟

### 5.2 tabBar 设计

tabBar 仅包含两个入口：

门店核销与管理后台通过角色分流进入，不占用 tabBar 位置。

## 6. 角色与权限体系

### 6.1 角色定义

系统定义 5 种角色，存储在 `portal_roles` 集合中：

| 角色键 | 角色名称 | 权限范围 | 典型权限 |
|--------|---------|---------|---------|
| `customer` | 客户 | 全局 | 查看本人合约、查看本人代金券、生成核销码 |
| `store_clerk` | 店员 | 单门店 | 查询核销码、确认核销、查看本店核销记录 |
| `store_owner` | 门店负责人 | 单门店 | 店员全部权限 + 查看本店经营数据 |
| `admin` | 管理员 | 全局 | 合约处理、门店管理、代金券规则管理、运营配置 |
| `super_admin` | 超级管理员 | 全局 | 管理员账号管理、角色分配与回收、全部权限 |

### 6.2 角色模型设计

```
┌─────────────────────────────────────────────────────┐
│                  portal_roles 集合                   │
├─────────────────────────────────────────────────────┤
│  openId      : 微信用户 openId                       │
│  roleKey     : 角色键（customer/store_clerk/...）    │
│  scopeType   : 权限范围类型（global/store/system）   │
│  scopeId     : 权限范围 ID（门店 ID，全局角色为空）  │
│  scopeName   : 范围名称（如门店名）                  │
│  name        : 角色显示名称                          │
│  phone       : 绑定手机号                            │
│  permissions : 权限列表（如 ['coupon.verify']）      │
│  status      : 状态（1=有效, 0=已禁用）              │
│  boundBy     : 绑定方式（auto_bind/self_bind/...）   │
│  createdAt   : 创建时间                              │
│  updatedAt   : 更新时间                              │
└─────────────────────────────────────────────────────┘
```

**关键设计：**

- **一个用户可有多个角色**：例如同一 openId 可以既是客户又是某门店负责人
- **scopeType + scopeId 精确控制范围**：门店角色绑定具体 storeId，管理员为全局
- **status 软删除**：禁用角色时设为 0，保留历史记录

### 6.3 角色识别与分流流程

```
[小程序启动]
    │
    ▼
[app.js onLaunch] ──初始化云开发──▶ [读取本地缓存的角色]
    │
    ▼
[pages/launch/index 启动页]
    │
    ▼
[调用 resolveLaunchContext 云函数]
    │
    ▼
[云函数查询 portal_roles where openId = 当前用户]
    │
    ├── 无角色 ──▶ 检查 staff_invites 待绑定邀请
    │               ├── 有邀请且手机号匹配 ──▶ 自动绑定 ──▶ 返回角色入口
    │               ├── 有邀请但需绑定 ──▶ 引导到 /pages/bind/index
    │               └── 无邀请 ──▶ 返回 customer 入口
    │
    ├── 单角色 ──▶ 直接返回该角色入口
    │
    └── 多角色 ──▶ 根据 lastPortalRole 记忆 / 返回角色选择列表
                    │
                    ▼
            [前端跳转到对应页面]
            ├── customer     ──▶ /pages/index/index
            ├── store_owner  ──▶ /pages/store/verify?storeId=xxx
            ├── store_clerk  ──▶ /pages/store/verify?storeId=xxx
            ├── admin        ──▶ /packageAdmin/pages/admin/index
            └── super_admin  ──▶ /packageAdmin/pages/admin/index
```

### 6.4 权限校验机制

权限校验**全部在云函数内执行**，前端不做任何权限判断（前端仅做界面展示控制）。云函数提供三个核心校验函数：

| 函数 | 位置 | 用途 |
|------|------|------|
| `ensureRole(openId, roleKeys)` | `utils/permission.js` + 各云函数内 | 校验用户是否具有指定角色之一 |
| `ensureSuperAdmin(openId)` | 同上 | 校验用户是否为超级管理员 |
| `ensureStoreRole(openId, storeId)` | 同上 | 校验用户是否具有指定门店的角色 |

**调用示例：**

```javascript
// 云函数内权限校验
async function handleAdminUpdateStatus(openId, event) {
  const isAdmin = await ensureRole(openId, ['admin', 'super_admin']);
  if (!isAdmin) {
    return { code: 403, message: '当前账号无管理员权限' };
  }
  // ... 业务逻辑
}
```

### 7.2 合约办理模块

**职责：** 提交手机号 → 资格核验 → 填写收货信息 → 验证码验证 → 合约完成 → 发货签收

**业务流程：**

```
[用户选择门店 + 输入手机号]
    │
    ▼ submitPhone
[创建合约记录，status=1 待核验]
    │
    ▼ 管理员审核（adminUpdateStatus）
[status=2 核验通过] 或 [status=0 拒绝]
    │
    ▼ submitOrderInfo（用户填写收货信息）
[status=3 待验证码]
    │
    ▼ submitSmsCode（用户回填验证码）
[status=4 办理中]
    │
    ▼ 管理员确认办理完成（adminUpdateStatus）
[status=5 合约已办] ──▶ 自动生成代金券（status=0 待激活）
    │
    ▼ 管理员填写快递单号
[status=6 已发货]
    │
    ▼ 用户签收 / 管理员确认
[status=7 已完成]
```

**关键设计：**

- **单合约约束**：用户同时只能有一个进行中的合约，已完成（status=5/6/7）后才允许新建
- **currentContractId 指针**：`portal_users.currentContractId` 记录用户当前合约，避免每次查询
- **验证码驳回机制**：status=8 验证码被驳回时，用户可重新提交

- `cloudfunctions/portalBiz/index.js`（`handleSubmitPhone`、`handleSubmitOrderInfo`、`handleSubmitSmsCode`、`handleAdminUpdateStatus`）
- `pages/index/index.js`（合约办理入口）
- `pages/contract/detail/index.js`（合约详情与进度）

### 7.3 权益管理模块

**职责：** 合约完成生成代金券 → 移动平台领取回调激活 → 30天有效期 → 到店核销 → 状态回传移动

**代金券生命周期：**

```
[合约 status=5 合约已办]
    │
    ▼ ensureCouponsForContract()
[创建代金券，status=0 待激活]
    │
    ▼ 用户去移动平台领取权益
    │
    ▼ 移动回调 handleActivateCoupon()
[代金券 status=1 可使用，计算30天有效期]
    │
    ▼ 用户在卡包生成核销码 generateVerifyCode()
[核销码有效期3分钟]
    │
    ├── 30天到期未使用 ──▶ status=3 已过期
    │
    └── 到店核销 ──▶ status=2 已核销 ──▶ 回传移动核销状态
```

**关键设计：**

- **代金券规则匹配**：根据 `coupon_rules` 集合的 `storeScope`（all/bound）和 `selectedStores` 匹配面额
- **30天有效期**：从移动回调激活时开始计算，而非合约完成时
- **核销码3分钟有效**：防止核销码被截图盗用
- **移动权益字段**：`mobileBenefitId`、`mobileOrderId`、`mobileNotifiedAt` 用于与移动平台对账

- `cloudfunctions/portalBiz/index.js`（`ensureCouponsForContract`、`handleActivateCoupon`、`handleGenerateVerifyCode`）

**核销流程：**

**关键文件：**

- `cloudfunctions/portalBiz/index.js`（`handleStoreVerifyCoupon`、`handleStoreConfirmVerify`、`handleGetStoreVerifyRecords`）
- `pages/store/verify.js`（门店核销工作台）

### 8.3 云函数统一规范

所有云函数遵循以下规范：

1. **统一返回格式**：`{ code: number, data?: any, message?: string }`
   - `code: 200` 成功
   - `code: 400` 参数错误
   - `code: 403` 权限不足
   - `code: 404` 资源不存在
   - `code: 429` 频率限制
   - `code: 500` 服务异常

2. **统一异常捕获**：`main` 函数外层 try-catch，避免云函数直接抛错

3. **统一初始化**：`cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })` 使用动态环境变量

4. **统一鉴权**：通过 `cloud.getWXContext()` 获取 OPENID，不信任前端传入的 openId

5. **统一频率限制**：调用 `checkRateLimit(OPENID, action)` 进行限流

### 10.2 对接接口

#### 10.2.1 移动回调激活代金券（移动 → 智机惠）

**接口：** 云函数 `portalBiz`，action=`activateCoupon`

**调用方：** 中国移动权益优选平台

**鉴权方式：** `authToken` 参数校验（当前为 `zjh_callback_2024`，正式上线需替换为移动约定的令牌）

**请求参数：**

```javascript
{

  contractId: 'C1234567890',     // 智机惠合约 ID
  mobileBenefitId: 'MB_xxx',     // 移动权益 ID
  mobileOrderId: 'MO_xxx',       // 移动订单 ID
  authToken: 'zjh_callback_2024' // 鉴权令牌
}
```

**处理逻辑：**

1. 校验 authToken
2. 查询该 contractId 下 status=0 的代金券
3. 更新代金券：status=0→1，记录 mobileBenefitId/mobileOrderId，计算30天有效期
4. 返回激活结果

**响应：**

```javascript
{
  code: 200,
  message: 'ok',
  data: {
    activatedCount: 1,
    contractId: 'C1234567890',
    expireAt: '2026-07-17T...'
  }
}
```

#### 10.2.2 核销状态回传（智机惠 → 移动）

**接口：** 待移动提供正式接口地址（当前为 TODO）

**触发时机：** 门店确认核销后（`handleStoreConfirmVerify`），异步调用 `notifyMobileVerifyStatus`

**回传内容（拟）：**

### 10.3 对接注意事项

1. **合约 ID 传递**：用户在移动平台领取权益时，需将智机惠的 `contractId` 传递给移动平台，移动回调时原样回传
2. **幂等性**：`activateCoupon` 接口需保证幂等，同一 contractId 多次回调只激活一次（通过查询 status=0 实现）
3. **核销回传容错**：核销状态回传失败不阻塞核销流程，仅记录日志，后续可通过定时任务补偿
4. **鉴权令牌**：`authToken` 当前硬编码在云函数中，建议迁移到云环境变量或移动约定的更安全方案

### 12.2 数据库安全

**所有数据库集合权限设置为：**

```json
{
  "read": false,
  "write": false
}
```

**含义：**

- 客户端（小程序前端）无法通过 `wx.cloud.database()` 直接读写任何集合
- 所有数据操作必须通过云函数 `wx.cloud.callFunction` 间接访问
- 云函数内通过 `cloud.database()` 拥有完整权限

**配置位置：** 微信云开发控制台 → 数据库 → 对应集合 → 权限设置 → 自定义安全规则

### 12.3 云函数权限校验

#### 12.3.1 角色校验函数

```javascript
// 校验用户是否具有指定角色之一
async function ensureRole(openId, roleKeys = []) {
  const res = await db.collection('portal_roles').where({
    openId,
    roleKey: _.in(roleKeys),
    status: 1
  }).limit(1).get();
  return !!(res.data && res.data.length);
}

// 校验用户是否为超级管理员
async function ensureSuperAdmin(openId) {
  const res = await db.collection('portal_roles').where({
    openId,
    roleKey: 'super_admin',
    status: 1
  }).limit(1).get();
  return !!(res.data && res.data.length);
}

// 校验用户是否具有指定门店的角色
async function ensureStoreRole(openId, storeId) {
  const res = await db.collection('portal_roles').where({
    openId,
    roleKey: _.in(['store_owner', 'store_clerk']),
    scopeType: 'store',
    scopeId: storeId,
    status: 1
  }).limit(1).get();
  return !!(res.data && res.data.length);
}
```

### 12.4 接口频率限制

基于 `rate_limits` 集合实现，按 `openId + action` 维度限流，1分钟滑动窗口：

| 操作类型 | 频率限制 | 说明 |
|---------|---------|------|
| 查询类（getStores, getCoupons 等） | 120/分钟 | 宽松，允许列表翻页 |
| 默认 | 60/分钟 | 兜底 |
| 写入类（submitPhone, submitOrderInfo 等） | 10/分钟 | 防止刷单 |
| 敏感操作（createNewContract, adminCreateStore） | 5/分钟 | 防止恶意创建 |
| 删除操作（adminDeleteStore, deleteCouponRule） | 3/分钟 | 防止误操作 |
| 身份绑定（bindRole） | 5/分钟 | 防止暴力绑定 |
| 短信发送（sendSmsCode） | 3/分钟 | 防止短信轰炸 |

**实现位置：** `cloudfunctions/rateLimit.js`（各云函数通过 `require('./rateLimit')` 引入）

### 12.5 参数验证

云函数内通过 `validateParams` 函数统一校验参数：

```javascript
function validateParams(params = {}, rules = {}) {
  // 支持：required(必填)、type(类型)、min/max(数值范围)
  //       minLength/maxLength(字符串长度)、pattern(正则)、message(错误提示)
}
```

**典型校验规则：**

- 手机号：`/^1[3-9]\d{9}$/`
- 验证码：`/^\d{6}$/`
- 必填字段：`required: true`
- 数值范围：`min: 0, max: 100`

### 12.6 日志脱敏

云函数内 `sanitizeForLog` 函数对敏感信息脱敏后再记录日志：

| 类型 | 脱敏规则 | 示例 |
|------|---------|------|
| phone | 前3后4，中间4位* | `138****1234` |
| idcard | 前4后4，中间* | `1101********1234` |
| name | 首字+* | `张**` |
| address | 前6后4，中间... | `河北省石家庄...1234` |
| default | 前3后3，中间* | `138***234` |

### 12.7 安全注意事项

1. **initDatabase 云函数必须删除**：上线前在云开发控制台删除 `initDatabase` 云函数，避免被恶意调用初始化数据
2. **authToken 必须更换**：移动回调的 `zjh_callback_2024` 为开发期占位值，正式上线前必须替换
3. **企业微信 Webhook 密钥保护**：`portalBiz/index.js` 中的 `WECOM_WEBHOOK` 含群机器人 key，避免泄露
4. **超级管理员 openId 保护**：`portal_roles` 中 super_admin 记录的 openId 是系统最高权限，避免泄露
5. **云函数日志查看权限**：云开发控制台日志可能含敏感信息，需控制访问权限

## 附录 A：技术决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 前端框架 | 微信原生 | 性能最优，包体积最小 |
| 后端架构 | 微信云开发 | 无服务器运维，与微信能力深度集成 |
| 数据库 | 云数据库（NoSQL） | 与云开发原生集成，schema-free 便于迭代 |
| UI 组件库 | Vant Weapp | 成熟稳定，组件丰富 |
| 角色识别 | 云函数查询 portal_roles | 不信任前端，权限收敛到云函数 |
| 频率限制 | 基于 rate_limits 集合 | 无需 Redis 等外部依赖 |
| 通知方式 | 企业微信 Webhook | 即时通知，无需开发通知系统 |

## 附录 B：待办与风险

### 待办事项

1. **移动核销状态回传**：`notifyMobileVerifyStatus` 函数待移动提供正式接口后补充实现
2. **authToken 安全升级**：将硬编码的 `zjh_callback_2024` 迁移到云环境变量或更安全方案
3. **企业微信 Webhook key 保护**：迁移到云环境变量
4. **initDatabase 云函数清理**：确认线上已删除
5. **订阅消息模板配置**：需在微信公众平台配置订阅消息模板

### 风险点

1. **单云函数过大**：`portalBiz` 承载了大部分业务逻辑，云函数包体积与冷启动时间需关注
2. **无数据备份**：云数据库需手动配置备份策略
3. **无监控告警**：建议配置云函数错误率告警
4. **移动对接依赖**：核销回传功能依赖移动提供接口，当前为 TODO 状态
