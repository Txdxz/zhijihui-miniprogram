```js
module.exports = {

  USE_CLOUD: true,

  // 填写在微信云开发控制台中的环境 ID
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `USE_CLOUD` | boolean | 是否启用云开发。`false` 时业务接口会提示"云开发未初始化" |

| `TRACE_USER` | boolean | 是否在云开发侧追踪用户访问行为 |

## 2. 部署步骤

### 2.1 本地开发环境搭建

#### 步骤 1：安装微信开发者工具

从微信官方下载并安装最新稳定版微信开发者工具。

#### 步骤 2：克隆代码

```bash
git clone https://github.com/Txdxz/zhijihui-miniprogram.git
cd zhijihui-miniprogram
```

#### 步骤 3：导入项目

1. 打开微信开发者工具
2. 点击"导入项目"
3. 项目目录选择克隆下来的仓库根目录
4. AppID 填写 `wxe3acd260b12d7843`（或使用"测试号"）
5. 后端服务选择"微信云开发"

#### 步骤 4：配置云环境 ID

编辑 `utils/cloud-env.js`，将 `CLOUD_ENV_ID` 改为你自己的云开发环境 ID：

```js
CLOUD_ENV_ID: 'cloud1-3goeo1mfba9edcd5'
```

> 如果是接手已有环境，直接使用上述 ID 即可。如果是新建环境，请在云开发控制台开通后替换。

#### 步骤 5：安装云函数依赖

进入每个云函数目录执行 `npm install`：

```bash
cd cloudfunctions/login && npm install
cd ../resolveLaunchContext && npm install
cd ../bindRoleByPhone && npm install
cd ../portalBiz && npm install
cd ../manageAdmin && npm install
cd ../initDatabase && npm install
```

> 本地安装依赖仅用于编辑器提示和本地调试。生产部署时勾选"云端安装依赖"即可。

#### 步骤 6：开通云开发环境

1. 在微信开发者工具顶部点击"云开发"按钮
2. 按提示开通环境
3. 记录环境 ID，回填到 `utils/cloud-env.js`

### 2.2 云函数部署

#### 部署方式

在微信开发者工具中，对每个云函数目录右键 → **"上传并部署：云端安装依赖"**。

#### 需要部署的云函数

| 序号 | 云函数路径 | 用途 | 是否常驻 |
|------|-----------|------|---------|
| 1 | `cloudfunctions/login` | 用户登录、建立会话、返回 openId | 是 |
| 2 | `cloudfunctions/resolveLaunchContext` | 识别用户角色、返回可进入的入口列表 | 是 |
| 3 | `cloudfunctions/bindRoleByPhone` | 根据手机号绑定管理员或门店角色 | 是 |
| 4 | `cloudfunctions/portalBiz` | 核心业务云函数（合约、卡券、核销、管理后台） | 是 |
| 5 | `cloudfunctions/mobileIntegration` | 世纪恒通荟采平台权益接入API（订单创建、查询、核销回调） | 是 |
| 6 | `cloudfunctions/notifyRetry` | 世纪恒通回传补推定时任务（每30分钟扫描未成功回传的券） | 是 |
| 7 | `cloudfunctions/manageAdmin` | 超管管理管理员邀请和角色回收 | 是 |
| 8 | `cloudfunctions/initDatabase` | 数据库初始化（**仅初始化时使用，上线前必须删除！**） | 否 |

#### 部署顺序建议

1. 先部署 `login`，验证登录链路
2. 再部署 `resolveLaunchContext`，验证角色识别
3. 再部署 `bindRoleByPhone` 和 `manageAdmin`，验证身份绑定
4. 再部署 `portalBiz`，验证业务流程
5. 再部署 `mobileIntegration`，配置 HTTP 触发器并与世纪恒通联调（详见 2.2.1）
6. 再部署 `notifyRetry`，配置定时触发器（详见 2.2.2）
7. 最后部署 `initDatabase`（仅用于初始化，详见 2.3）

#### 部署后验证

每个云函数部署完成后，在云开发控制台 → 云函数 → 对应函数 → "云端测试"中调用，确认返回正常。

### 2.2.1 mobileIntegration HTTP 触发器配置

`mobileIntegration` 云函数需要配置 HTTP 触发器，以便世纪恒通荟采平台通过 HTTP 方式调用智机惠的权益接入 API（订单创建、查询、核销回调）。

#### 配置步骤

1. 进入 **微信云开发控制台 → 云函数 → `mobileIntegration` → 触发器**
2. 创建 HTTP 触发器，路径填写如 `/mobileIntegration`
3. 触发器创建完成后，会生成对外访问 URL，格式为：

   ```
   https://{云环境ID}.service.tcloudbasegateway.com/mobileIntegration/order/create
   ```

   其中 `{云环境ID}` 为 `utils/cloud-env.js` 中的 `CLOUD_ENV_ID`（当前为 `cloud1-3goeo1mfba9edcd5`）。

4. 将上述完整 URL 提供给世纪恒通，作为其调用智机惠 API 的接口地址

> ⚠️ HTTP 触发器地址为对外暴露的接口，必须确保 `mobileIntegration` 内部的鉴权（appId + appSecret + MD5 签名）和加解密（3DES）逻辑已正确配置，避免被未授权调用。

#### mobileIntegration 配置项

部署 `mobileIntegration` 前，需先编辑 `cloudfunctions/mobileIntegration/config.js` 完成以下配置：

| 配置项 | 提供方 | 说明 |
|--------|--------|------|
| `appId` | 智机惠 | 智机惠分配给世纪恒通的应用 ID，需分享给世纪恒通 |
| `appSecret` | 智机惠 | 智机惠分配给世纪恒通的应用密钥，需分享给世纪恒通 |
| `desKey` | 世纪恒通 | 世纪恒通提供的 24 位 3DES 密钥，用于请求体加解密 |
| `hengtongApiBase` | 世纪恒通 | 世纪恒通 API 域名（如需要回调世纪恒通时使用） |

> 详细对接说明见 `docs/handover/06-世纪恒通API对接说明.md`。

### 2.2.2 notifyRetry 定时触发器配置

`notifyRetry` 云函数用于定时扫描已核销/已自动结算但未成功回传世纪恒通的券，重新调用回传逻辑，满足API文档2.2节"状态补推"要求。

#### 配置步骤

1. 进入 **微信云开发控制台 → 云函数 → `notifyRetry` → 触发器**
2. 创建定时触发器，Cron 表达式填写 `0 */30 * * * * *`（每30分钟执行一次）
3. 触发器名称填写如 `notifyRetryTimer`

#### 配置项

部署 `notifyRetry` 前，需先编辑 `cloudfunctions/notifyRetry/config.js`，配置项与 `mobileIntegration/config.js` **保持一致**：

| 配置项 | 说明 |
|--------|------|
| `appId` | 与 `mobileIntegration/config.js` 相同 |
| `appSecret` | 与 `mobileIntegration/config.js` 相同 |
| `desKey` | 与 `mobileIntegration/config.js` 相同 |
| `hengtongApiBase` | 与 `mobileIntegration/config.js` 相同 |

> ⚠️ 两个云函数的 `config.js` 必须保持一致，否则签名/加解密会失败。建议联调时同步更新。

#### 执行逻辑

1. 查询 `coupons` 集合中 `settleType` 为 `auto` 或 `manual`、`mobileNotifiedAt` 为空、`notifyUrl` 不为空的券（每批最多50条）
2. 对每张券调用回传逻辑，单券重试2次，间隔2秒/4秒
3. 回传成功后写入 `mobileNotifiedAt` 字段
4. 返回执行摘要（扫描数、成功数、失败数）

#### 监控建议

- 定期查看 `notifyRetry` 云函数日志，关注失败数
- 若失败数持续较高，检查世纪恒通接口可用性和密钥配置
- 回传失败超过3次的券需人工排查

### 2.3 数据库初始化

#### 步骤 1：部署 initDatabase 云函数

在微信开发者工具中，右键 `cloudfunctions/initDatabase` → **"上传并部署：云端安装依赖"**。

#### 步骤 2：调用 initDatabase 云函数

在云开发控制台 → 云函数 → `initDatabase` → "云端测试"中，输入以下参数：

```json
{
  "initSecret": "zjh_init_2024"

```

> ⚠️ `initSecret` 是初始化密钥，必须正确传入才能执行。错误密钥会返回 `code: 403`。

#### 步骤 3：验证初始化结果

`initDatabase` 会自动完成以下工作：

| 添加 7 家门店 | 中国移动权益指定门店（store_001 ~ store_007，均位于河北省） |
| 创建数据库索引 | 为各集合的关键字段创建索引，提升查询性能 |

验证方法：

1. 云开发控制台 → 数据库，确认 10 个集合均已创建
2. 打开 `portal_roles` 集合，确认有一条 `roleKey: "super_admin"` 记录
3. 打开 `stores` 集合，确认有 7 条门店记录（store_001 ~ store_007）
4. 检查各集合的"索引管理"，确认索引已创建

#### 步骤 4：删除 initDatabase 云函数（重要！）

> ⚠️ **安全要求：初始化完成后必须立即删除 `initDatabase` 云函数！**
>
> 删除方法：在微信开发者工具中右键 `cloudfunctions/initDatabase` → "删除"，或在云开发控制台 → 云函数 → `initDatabase` → "删除"。

**删除原因：**

- 避免被恶意调用重复创建数据
- `initDatabase` 内部虽有 `initSecret` 校验和"已存在超管则禁止重复初始化"的保护，但仍不应长期暴露
- 符合最小权限暴露原则

### 2.4 数据库权限设置

> ⚠️ **所有集合必须设置为自定义安全规则：`{ "read": false, "write": false }`**

#### 设置步骤

1. 云开发控制台 → 数据库 → 点击对应集合
2. 点击"权限设置"标签
3. 选择"自定义安全规则"
4. 粘贴以下规则：

```json
{
  "read": false,
  "write": false
}
```

5. 保存

#### 权限说明

- `read: false`：禁止客户端直接读取，所有数据访问必须通过云函数
- `write: false`：禁止客户端直接写入，所有数据操作必须通过云函数

#### 需要设置的集合清单

以下 10 个集合均需按上述规则设置：

```
portal_users、portal_roles、stores、contracts、coupons、
staff_invites、coupon_rules、login_sessions、
sms_verifications、rate_limits
```

> ⚠️ 权限设置错误会导致客户端可直读直写数据库，存在严重安全风险。每次环境变更后必须复核。

### 2.5 超级管理员配置

#### 方式一：通过 initDatabase 自动配置（推荐）

调用 `initDatabase` 云函数时，**第一个调用者**会自动成为 `super_admin`。

#### 方式二：手动添加

如果需要新增其他超级管理员，或在不初始化的情况下手动配置：

1. 让目标用户先打开小程序并完成登录（触发 `login` 云函数）
2. 在云开发控制台 → 数据库 → `portal_roles` 集合 → "添加记录"，切换到 JSON 模式，粘贴：

```json
{
  "openId": "目标用户的 openId",
  "roleKey": "super_admin",
  "status": 1,
  "scopeType": "global",
  "scopeId": "",
  "createdAt": "2026-04-10T08:00:00.000Z",
  "updatedAt": "2026-04-10T08:00:00.000Z"
}
```

3. 保存后，目标用户重新进入小程序即可进入管理后台

#### 如何获取 openId

1. 让目标用户打开小程序
2. 小程序会自动调用 `wx.cloud.callFunction({ name: 'login' })`
3. 在云开发控制台 → 云函数 → `login` → 日志中查看返回的 `openId`

#### 角色权限说明

| 角色 | roleKey | 典型权限 |
|------|---------|---------|
| 客户 | `customer` | 查看本人合约、卡包、生成核销码 |
| 店员 | `store_clerk` | 查询核销码、确认核销、查看本店核销记录 |
| 门店负责人 | `store_owner` | 具备店员权限，并可查看本店经营数据 |
| 管理员 | `admin` | 合约处理、代金券规则管理、门店管理 |
| 超级管理员 | `super_admin` | 管理管理员账号、分配角色、回收角色 |

> ⚠️ `admin` 和 `super_admin` 必须区分。只有 `super_admin` 才能管理管理员账号，避免权限滥用。

1. 审核通过后，在"版本管理"中找到已通过版本
2. 点击"发布"
3. 确认发布

发布后必须确认：

- [ ] 生产环境数据库集合与开发环境一致（10 个集合）
- [ ] 所有集合权限为 `{ "read": false, "write": false }`

- [ ] `utils/cloud-env.js` 的 `CLOUD_ENV_ID` 指向生产环境
- [ ] `initDatabase` 云函数已删除
- [ ] 超级管理员已配置
- [ ] 门店数据已录入

## 3. 日常运维

### 3.1 监控要点

#### 云函数日志

- **入口**：云开发控制台 → 云函数 → 选择函数 → "日志"
- **关注点**：
  - `console.error` 输出（错误日志）
  - 调用耗时（P99 是否异常升高）
  - 调用失败率
- **重点函数**：`portalBiz`（业务核心）、`login`（登录入口）

#### 数据库慢查询

- **入口**：云开发控制台 → 数据库 → "慢查询"
- **关注点**：
  - 查询耗时超过 100ms 的记录
  - 是否缺少索引（参考 2.3 节索引清单）
  - 是否存在全表扫描

#### 云函数调用频率

- **入口**：云开发控制台 → 云函数 → "监控"
- **关注点**：
  - 调用量突增（可能是异常或攻击）
  - 并发数是否接近上限
  - 频率限制（`rateLimit.js`）是否被频繁触发

#### 错误告警

- **关注点**：所有 `console.error` 输出
- **建议**：定期（每日）巡查云函数日志中的 ERROR 级别记录
- **企业微信通知**：`portalBiz` 已接入企业微信群机器人，关键业务事件（新客户申请、验证码提交等）会自动推送

#### 监控建议频率

| 监控项 | 建议频率 |
|--------|---------|
| 云函数错误日志 | 每日 1 次 |
| 数据库慢查询 | 每周 1 次 |
| 云函数调用量 | 每周 1 次 |
| 数据库容量 | 每月 1 次 |
| 云函数版本一致性 | 每次发布前后 |

### 3.2 常见问题处理

#### Q1：用户无法登录

**排查步骤：**

1. **检查 `cloud-env.js` 配置**
   - `USE_CLOUD` 是否为 `true`
   - `CLOUD_ENV_ID` 是否与云开发控制台一致
2. **检查 `login` 云函数**
   - 是否已部署
   - 云开发控制台 → 云函数 → `login` → 日志，查看是否有报错
3. **检查 `portal_users` 集合**
   - 权限是否为 `{ "read": false, "write": false }`
   - 集合是否存在
4. **检查网络**
   - 用户网络是否正常
   - 微信开发者工具中是否有"云开发未初始化"提示

#### Q2：管理员无法进入后台

**排查步骤：**

1. **检查 `portal_roles` 集合**
   - 是否有该用户 `openId` 对应的 `admin` 或 `super_admin` 记录
   - `status` 是否为 `1`（启用）
2. **检查 `resolveLaunchContext` 云函数**
   - 云开发控制台 → 云函数 → `resolveLaunchContext` → 日志
   - 确认返回的角色列表是否包含 `admin` 或 `super_admin`
3. **重新登录**
   - 让用户退出小程序重新进入，触发角色重新识别

#### Q3：代金券无法核销

**排查步骤：**

1. **检查券状态**
   - 在 `coupons` 集合中查看该券的 `status`
   - 必须为 `1`（可使用）才能核销
   - 状态码：`1`=可使用，`2`=已核销，`3`=已过期
2. **检查是否过期**
   - 查看 `expireAt` 字段，是否已超过当前时间
   - 过期券需联系管理员处理
3. **检查门店权限**
   - 在 `portal_roles` 中确认核销人是否有该门店的 `store_owner` 或 `store_clerk` 角色
   - `scopeId` 是否与券的 `storeId` 一致
4. **检查核销码**
   - `verifyCode` 是否正确
   - `verifyExpireAt` 是否已过期（核销码有效期 3 分钟）

#### Q4：世纪恒通 /order/create 调用失败

**排查步骤：**

1. **检查签名验证**
   - 世纪恒通请求 Header 中的 `appId`、`sign` 是否正确
   - 签名算法：appId + traceId + timestamp + nonceStr + 业务参数(字典排序) + appSecret → MD5
   - 错误的签名会返回 `code: 401, message: "签名验证失败"`
2. **检查 3DES 加解密**
   - `config.js` 中的 `desKey` 是否与世纪恒通一致（24位）
   - 请求体 `ciphertext` 字段是否能正确解密
3. **检查必填参数**
   - `orderNo`、`rechargeAccount`（手机号）、`skuId`、`notifyUrl` 是否齐全
4. **查看云函数日志**
   - 云开发控制台 → 云函数 → `mobileIntegration` → 日志
   - 搜索 `/order/create` 相关记录

#### Q5：企业微信通知不工作

**排查步骤：**

1. **检查 Webhook 地址**
   - 打开 `cloudfunctions/portalBiz/index.js`
   - 查找 `WECOM_WEBHOOK` 常量
   - 确认地址是否有效（企业微信群机器人地址）
2. **测试 Webhook**
   - 复制 Webhook 地址，用 curl 或 Postman 手动发送测试消息
   - 确认机器人是否被禁用或删除
3. **检查云函数日志**
   - 搜索 `wecom notify error`
   - 查看具体错误信息（网络问题、地址失效等）
4. **更新 Webhook**
   - 如机器人地址失效，在企业微信群中重新添加机器人
   - 更新 `portalBiz/index.js` 中的 `WECOM_WEBHOOK`
   - 重新部署 `portalBiz` 云函数

- **删除方法**：
  - 微信开发者工具中右键 `cloudfunctions/initDatabase` → "删除"

- **验证**：删除后，在云开发控制台 → 云函数列表中确认 `initDatabase` 已不存在

> ⚠️ **删除云函数，不是注释代码！** 必须从云端彻底删除。

### 4.4 接口安全

#### 频率限制

- **实现位置**：`cloudfunctions/rateLimit.js`（被 `portalBiz`、`bindRoleByPhone`、`manageAdmin` 引用）
- **存储集合**：`rate_limits`
- **限制规则**：

| 操作类型 | 频率限制 |
|---------|---------|
| 查询类（getStores、getCoupons 等） | 每分钟 120 次 |
| 默认 | 每分钟 60 次 |
| 写入类（submitPhone、submitOrderInfo 等） | 每分钟 10 次 |
| 敏感操作（createNewContract、adminCreateStore） | 每分钟 5 次 |
| 删除操作（adminDeleteStore、deleteCouponRule） | 每分钟 3 次 |

- **超限返回**：`code: 429, message: "操作过于频繁，请 X 秒后重试"`

#### 世纪恒通 API 鉴权

- **接口**：`mobileIntegration` 云函数（HTTP 触发）
- **鉴权方式**：appId + appSecret + MD5 签名（32位小写）
- **加密方式**：3DES-ECB / PKCS5Padding / hex 输出
- **配置文件**：`cloudfunctions/mobileIntegration/config.js`
- **失败返回**：`code: 401, message: "签名验证失败"`

#### 管理接口鉴权

- **实现方式**：每个管理类 action 在云函数内校验调用者角色
- **校验逻辑**：
  - `admin` 相关操作：校验 `admin` 或 `super_admin` 角色
  - `super_admin` 专属操作（如管理管理员、创建门店）：仅校验 `super_admin` 角色
  - 门店核销：校验 `store_owner` 或 `store_clerk` 角色，且 `scopeId` 与门店一致
- **失败返回**：`code: 403, message: "当前账号无 XX 权限"`

#### 安全审计建议

- 定期检查 `rate_limits` 集合，关注是否有异常高频调用
- 关注 `portal_roles` 集合变更，防止未授权角色提升
- 关键操作（核销、状态变更）建议补充审计日志集合 `audit_logs`

## 附录 C：代金券状态码

| 状态码 | 含义 | 说明 |
|--------|------|------|
| `1` | 可使用 | 世纪恒通 /order/create 生成后激活，可生成核销码并到店核销 |
| `2` | 已核销 | 已在门店核销使用 |
| `3` | 已过期 | 超过 `expireAt` 时间，系统自动标记 |

### 代金券有效期规则

- **当月有效**（领取到账之日起到当月最后一天 23:59:59）
- 由 `mobileIntegration` 的 `/order/create` 在生成券时计算并写入 `expireAt`
- 核销或生成核销码时会校验 `expireAt`，过期券自动标记为 `3`

## 附录 D：合约状态码

| 状态码 | 含义 | 说明 |
|--------|------|------|
| `0` | 已拒绝 | 申请被驳回 |
| `1` | 待核验 | 用户已提交手机号，等待管理员核验 |
| `2` | 核验通过 | 管理员核验通过 |
| `3` | 待验证码 | 用户已填写收货信息，等待提交验证码 |
| `4` | 办理中 | 用户已提交验证码，等待合约办理完成 |
| `5` | 合约已办 | 合约办理完成，自动生成代金券 |
| `6` | 已发货 | 已发货 |
| `7` | 已完成 | 已签收完成 |
| `8` | 验证码被驳回 | 验证码无效或已过期，需用户重新提交 |

### 合约状态流转

```
待核验(1) ──核验通过──→ 核验通过(2) ──填写收货──→ 待验证码(3)
   │                                                        │
   │拒绝                                                     │提交验证码
   ↓                                                        ↓
已拒绝(0)                                                办理中(4)
                                                            │
                                                            │合约办妥
                                                            ↓
                                                       合约已办(5) ──→ 自动生成代金券
                                                            │
                                                            │发货
                                                            ↓
                                                       已发货(6)
                                                            │
                                                            │签收
                                                            ↓
                                                       已完成(7)

办理中(4) ──验证码驳回──→ 验证码被驳回(8) ──重新提交──→ 办理中(4)
```
