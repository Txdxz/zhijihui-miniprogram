## 1. 概述

智机惠小程序基于**微信云开发**构建，所有数据读写均通过云函数完成，前端不直接操作数据库。本文档描述全部 6 个云函数及其 30+ 个接口的请求参数、返回格式、权限要求和业务逻辑。

### 云函数清单

| 云函数 | 职责 | 接口数量 |
|--------|------|----------|
| `login` | 用户登录，获取 openId | 1 |
| `resolveLaunchContext` | 启动时解析用户角色与跳转目标 | 1 |
| `bindRoleByPhone` | 通过手机号绑定角色 | 1 |
| `manageAdmin` | 超级管理员管理管理员账号 | 4 |
| `portalBiz` | 核心业务（合约、代金券、门店、管理等） | 30 |
| `initDatabase` | 数据库初始化（仅开发环境） | 1 |

## 2. 统一约定

### 2.1 调用方式

```javascript
wx.cloud.callFunction({
  name: '<云函数名>',
  data: { /* 参数 */ }
}).then(res => {
  console.log(res.result); // 统一返回格式
});
```

- 所有云函数通过 `cloud.getWXContext()` 自动获取调用者的 `OPENID`，**无需前端传递**。
- `portalBiz` 云函数通过 `action` 字段区分不同接口。

### 2.2 统一返回格式

```typescript
{
  code: number,       // 状态码
  message?: string,   // 错误描述（仅非 200 时必返回）
  data?: any          // 业务数据（仅 200 时返回）
}
```

### 2.3 错误码

| code | 含义 | 典型场景 |
|------|------|----------|
| 200 | 成功 | — |
| 400 | 参数错误 / 业务规则不满足 | 缺少必填字段、状态不允许操作 |
| 403 | 权限不足 | 非管理员调用管理接口、authToken 验证失败 |
| 404 | 记录不存在 | 合约/门店/代金券未找到 |
| 429 | 频率限制 | 同一用户短时间内调用过多 |
| 500 | 服务异常 | 云函数内部错误 |

### 2.4 频率限制

- 通过 `rateLimit.js` 实现，基于云数据库 `rate_limits` 集合记录调用次数。
- 以 **openId + action** 为维度，在 **1 分钟滑动窗口** 内计数。
- 不同 action 有不同的频率上限（详见[附录 D](#附录-d频率限制配置表)）。
- 超限时返回 `{ code: 429, message: "操作过于频繁，请 N 秒后重试" }`。

### 2.5 权限体系

| 角色标识 | 说明 | 典型权限 |
|----------|------|----------|
| `super_admin` | 超级管理员 | 全部权限，含门店 CRUD、代金券规则管理、管理员管理 |
| `admin` | 管理员 | 合约管理、门店查看、统计、导出 |
| `store_owner` | 门店负责人 | 门店核销、核销记录查看 |
| `store_clerk` | 店员 | 门店核销、核销记录查看 |
| `customer` | 普通客户（默认） | 提交合约、查看代金券、生成核销码 |

> 权限校验在云函数内部完成，前端仅做界面控制。

## 3. login — 登录

获取用户 openId，创建或更新 `portal_users` 记录。

### 请求参数

无。

### 返回示例

```json
{
  "code": 200,
  "data": {
    "openId": "oXXXXXXXXXXXXXXXX",
    "user": {
      "_id": "abc123",
      "openId": "oXXXXXXXXXXXXXXXX",
      "nickName": "",
      "avatarUrl": "",
      "phone": "",
      "status": 1,
      "lastLoginAt": "2026-06-17T08:00:00.000Z"
    }
  }
}
```

### 业务逻辑

1. 通过 `cloud.getWXContext()` 获取 `OPENID` 和 `APPID`。
2. 查询 `portal_users` 集合：
   - **不存在** → 创建新用户记录（nickName/avatarUrl/phone 为空，status=1）。
   - **已存在** → 更新 `lastLoginAt` 和 `updatedAt`。
3. 返回 openId 和完整用户信息。

## 4. resolveLaunchContext — 启动上下文解析

小程序启动时调用，根据用户角色返回跳转目标和可切换的角色入口。

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| lastPortalRole | string | 否 | 上次选择的角色标识，用于记忆用户偏好 |

### 返回示例

**场景 1：普通客户（无角色）**

```json
{
  "code": 200,
  "data": {
    "openId": "oXXX",
    "role": "customer",
    "target": {
      "key": "customer",
      "title": "客户首页",
      "desc": "办理合约与查看权益",
      "url": "/pages/index/index",
      "routeType": "switchTab"
    },
    "roleOptions": [],
    "needsChoice": false
  }
}
```

**场景 2：有待绑定邀请，引导绑定**

```json
{
  "code": 200,
  "data": {
    "openId": "oXXX",
    "role": "bind",
    "target": {
      "key": "bind",
      "title": "身份绑定",
      "desc": "检测到管理员邀请，请绑定手机号完成身份认证",
      "url": "/pages/bind/index?inviteType=admin",
      "routeType": "reLaunch",
      "inviteType": "admin"
    },
    "roleOptions": [],
    "needsChoice": false,
    "needsPhoneBind": true,
    "pendingInvite": {
      "roleKey": "admin",
      "name": "张三"
    }
  }
}
```

**场景 3：多角色需选择**

```json
{
  "code": 200,
  "data": {
    "openId": "oXXX",
    "role": "",
    "target": null,
    "roleOptions": [
      { "key": "customer", "title": "客户首页", "desc": "办理合约与查看权益", "url": "/pages/index/index", "routeType": "switchTab" },
      { "key": "admin", "title": "管理后台", "desc": "合约、门店与运营配置", "url": "/packageAdmin/pages/admin/index", "routeType": "reLaunch" }
    ],
    "needsChoice": true
  }
}
```

### 返回字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| openId | string | 当前用户 openId |
| role | string | 当前推荐角色（空字符串表示需选择） |
| target | object\|null | 推荐跳转目标，null 表示需用户选择 |
| target.key | string | 角色标识：customer / admin / super_admin / store_owner / store_clerk / bind |
| target.title | string | 入口标题 |
| target.desc | string | 入口描述 |
| target.url | string | 跳转页面路径 |
| target.routeType | string | 跳转方式：switchTab / reLaunch |
| target.inviteType | string | 仅 bind 类型存在，邀请角色类型 |
| target.storeId | string | 仅门店角色存在，关联门店 ID |
| roleOptions | array | 可切换的角色入口列表（含客户首页） |
| needsChoice | boolean | 是否需要用户手动选择角色 |
| needsPhoneBind | boolean | 是否需要绑定手机号（仅 bind 场景） |
| pendingInvite | object | 待绑定邀请信息（仅 bind 场景） |

### 业务逻辑

1. 确保 `portal_users` 记录存在（不存在则创建）。
2. 查询 `portal_roles` 中该用户所有 status=1 的角色。
3. 查询关联门店信息构建门店角色入口。
4. **自动绑定逻辑**：若用户无角色但有待绑定邀请（`staff_invites` 中 boundOpenId 为空），且用户手机号与邀请手机号匹配，则自动创建角色并标记邀请已绑定。
5. 根据 `lastPortalRole` 参数和角色数量决定返回策略：
   - 无角色 + 无邀请 → 直接进入客户首页
   - 无角色 + 有邀请 → 引导绑定
   - 单角色 → 直接进入
   - 多角色 + lastPortalRole 匹配 → 进入上次角色
   - 多角色 + 不匹配 → 返回 needsChoice=true 让用户选择

## 5. bindRoleByPhone — 手机号绑定角色

用户在绑定页面输入手机号，系统查找匹配的待绑定邀请并创建角色。

### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| phone | string | 是 | 手机号，11 位，1 开头 |
| roleKey | string | 是 | 角色标识：admin / store_owner / store_clerk |

### 返回示例

**成功**

```json
{
  "code": 200,
  "data": {
    "openId": "oXXX",
    "roleKey": "admin",
    "scopeType": "system",
    "scopeId": "portal"
  }
}
```

**失败 — 手机号格式错误**

```json
{ "code": 400, "message": "请输入正确的11位手机号" }
```

**失败 — 无可绑定邀请**

```json
{ "code": 404, "message": "未找到可绑定的角色，请联系管理员配置" }
```

### 业务逻辑

1. 验证手机号格式（`/^1[3-9]\d{9}$/`）和 roleKey 合法性。
2. 在 `staff_invites` 中查找 phone 匹配、roleKey 匹配、status=1、boundOpenId 为空的邀请。
3. 检查是否已有相同角色记录（防止重复绑定）。
4. 创建 `portal_roles` 记录（boundBy='self_bind'）。
5. 标记邀请为已绑定（填充 boundOpenId 和 boundAt）。

### 频率限制

- `bindRole`：每分钟 5 次

## 6. manageAdmin — 管理员管理

超级管理员管理管理员账号的邀请、信息更新和停用。

> **权限要求**：所有 action 均需 `super_admin` 角色。

### 6.1 action=list — 管理员列表

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| action | string | 是 | 固定值 `list` |

#### 返回示例

```json
{
  "code": 200,
  "data": [
    {
      "_id": "role001",
      "openId": "oXXX",
      "roleKey": "admin",
      "name": "张三",
      "phone": "13800138000",
      "scopeType": "system",
      "scopeId": "portal",
      "status": 1
    }
  ]
}
```

#### 业务逻辑

1. 查询 `portal_roles` 中 roleKey 为 admin 或 super_admin 且 status=1 的记录。
2. 从 `staff_invites` 补充 phone 和 name 信息（兼容旧数据）。

---

### 6.2 action=invite — 邀请管理员

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| action | string | 是 | 固定值 `invite` |
| phone | string | 是 | 被邀请人手机号 |
| name | string | 是 | 被邀请人姓名 |

#### 返回示例

```json
{
  "code": 200,
  "data": {
    "_id": "inv001",
    "phone": "13800138000",
    "name": "张三",
    "roleKey": "admin",
    "scopeType": "system",
    "scopeId": "portal",
    "scopeName": "智机惠管理后台",
    "permissions": ["contract.manage", "store.manage", "coupon.rules.manage"],
    "status": 1,
    "boundOpenId": "",
    "createdBy": "oYYY"
  }
}
```

#### 业务逻辑

1. 验证 name 非空、phone 格式正确。
2. 检查是否已有相同手机号的管理员邀请：
   - **已有** → 更新 name 和 updatedAt。
   - **没有** → 创建新的 `staff_invites` 记录，permissions 默认为 `['contract.manage', 'store.manage', 'coupon.rules.manage']`。

---

### 6.3 action=update — 更新管理员信息

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| action | string | 是 | 固定值 `update` |
| roleId | string | 是 | portal_roles 记录的 _id |
| name | string | 是 | 管理员姓名 |
| phone | string | 否 | 手机号（如提供需格式正确） |

#### 返回示例

```json
{
  "code": 200,
  "data": {
    "roleId": "role001",
    "name": "张三",
    "phone": "13800138000",
    "updatedAt": "2026-06-17T08:00:00.000Z"
  }

```

#### 业务逻辑

1. 更新 `portal_roles` 中对应记录的 name、phone、updatedAt。
2. 若 phone 变更，同步更新 `staff_invites` 中对应记录。

---

### 6.4 action=disableRole — 停用管理员角色

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| action | string | 是 | 固定值 `disableRole` |
| roleId | string | 是 | portal_roles 记录的 _id |

#### 返回示例

```json
{
  "code": 200,
  "data": { "roleId": "role001", "status": 0 }
}
```

#### 业务逻辑

1. 检查目标角色是否为 super_admin，是则拒绝操作。
2. 将 `portal_roles` 中对应记录的 status 设为 0（软删除）。

#### 7.3.0b tryAutoSettle — 懒触发自动结算（内部函数）

**说明**：`portalBiz` 云函数内部函数，在代金券相关操作（查询、生成核销码等）时懒触发，检查是否有到期未结算的券，自动回传结算状态给中国移动。

**触发时机**：`getCoupons`、`generateVerifyCode` 等接口调用时自动触发。

**业务逻辑**

1. 查询当前用户所有 status=2（已核销）且 `settleType` 不为 `auto` 的代金券。
2. 检查 `expireAt` 是否已到期。
3. 到期券调用 `notifyMobileSettle` 回传结算状态。
4. 回传成功后标记 `settleType='auto'`、`settleAt` 为当前时间。

**注意**：此函数为内部调用，不对外暴露为独立 action。

## 附录 A：合约状态流转

| 状态码 | 常量名 | 含义 | 可转换到 |
|--------|--------|------|----------|
| 0 | REJECTED | 已拒绝 | — |
| 1 | WAIT_VERIFY | 待核验 | 0, 2 |
| 2 | QUALIFIED | 核验通过 | 3 |
| 3 | WAIT_SMSCODE | 待验证码 | 4, 8 |
| 4 | CONTRACTING | 办理中 | 5, 8 |
| 5 | CONTRACT_OK | 合约已办 | 6 |
| 6 | SHIPPED | 已发货 | 7 |
| 7 | SIGNED | 已完成（签收） | — |
| 8 | SMS_CODE_REJECTED | 验证码被驳回 | 4 |

**典型流转路径**：

```
1(待核验) → 2(核验通过) → 3(待验证码) → 4(办理中) → 5(合约已办) → 6(已发货) → 7(已完成)
                                    ↑                    ↓
                                    └── 8(验证码被驳回) ──┘
```
